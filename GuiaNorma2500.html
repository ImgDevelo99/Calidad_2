Paso 1: Introducción a ISO/IEC 25000 (SQuaRE)

¡Bienvenido a esta guía interactiva sobre la familia de normas ISO/IEC 25000, también conocida como SQuaRE (System and Software Quality Requirements and Evaluation)! A nivel intermedio, profundizaremos en cómo estas normas nos ayudan a entender, especificar y evaluar la calidad del software y los sistemas.

SQuaRE reemplaza a las antiguas normas ISO 9126 e ISO 14598, proporcionando un marco más completo y moderno para la gestión de la calidad del producto de software.

En este módulo, nos enfocaremos principalmente en dos partes clave de la familia:

  • ISO/IEC 25010: Modelo de Calidad de Producto (El 'qué' de la calidad).
  • ISO/IEC 2502x: Modelos de Medición de Calidad (El 'cómo' medir la calidad).

Paso 2: ISO/IEC 25010 - El Modelo de Calidad de Producto

ISO/IEC 25010 define un modelo de calidad que categoriza las propiedades de calidad de un producto de software. Este modelo es jerárquico y se divide en 8 características principales, cada una subdividida en sub-características.

Entender este modelo es fundamental para especificar y evaluar la calidad de manera estructurada.

Las 8 características principales son:

  1. Adecuación funcional (Functional Suitability)
  2. Eficiencia de desempeño (Performance Efficiency)
  3. Compatibilidad (Compatibility)
  4. Usabilidad (Usability)
  5. Fiabilidad (Reliability)
  6. Seguridad (Security)
  7. Mantenibilidad (Maintainability)
  8. Portabilidad (Portability)

Cada característica se descompone en sub-características que describen aspectos más específicos de la calidad.

Paso 3: Detalle de las Características de ISO/IEC 25010

Veamos algunas de las características con más detalle, enfocándonos en qué implica cada una y sus sub-características clave.

Fiabilidad (Reliability)

Mide la capacidad del sistema para desempeñar sus funciones especificadas en condiciones determinadas durante un período de tiempo determinado.

  • Madurez (Maturity): Frecuencia de fallos.
  • Disponibilidad (Availability): Capacidad de estar operativo y accesible cuando se necesita.
  • Tolerancia a fallos (Fault Tolerance): Capacidad de funcionar a pesar de la presencia de fallos.
  • Capacidad de recuperación (Recoverability): Capacidad de restablecer un estado operativo y recuperar datos tras un fallo.

Usabilidad (Usability)

Mide la capacidad del producto para ser entendido, aprendido, operado y atractivo para el usuario, cuando es usado en condiciones específicas.

  • Capacidad para ser entendido (Understandability): Facilidad con la que los usuarios reconocen el propósito y uso del producto.
  • Capacidad de aprendizaje (Learnability): Facilidad con la que los usuarios pueden aprender a usar el producto.
  • Capacidad para ser operado (Operability): Facilidad con la que los usuarios pueden operar y controlar el producto.
  • Protección frente a errores de usuario (User Error Protection): Capacidad de prevenir errores de usuario o mitigar sus consecuencias.
  • Estética de la interfaz de usuario (User Interface Aesthetics): Qué tan agradable y atractiva es la interfaz.
  • Accesibilidad (Accessibility): Facilidad con la que el producto puede ser usado por personas con diversas capacidades.

Y así con cada una de las 8 características...

Paso 4: ISO/IEC 2502x - Modelos de Medición de Calidad

El modelo 25010 nos dice 'qué' es la calidad. Las normas 25020, 25021, etc., nos dicen 'cómo' medir esa calidad.

La medición de la calidad implica definir métricas que permitan cuantificar las características y sub-características.

Conceptos clave:

  • Medida (Measure): Una variable a la que se asigna un valor numérico (ej: número de fallos, tiempo de respuesta).
  • Medida Base (Base Measure): No depende de otra medida (ej: número de líneas de código, número de fallos detectados).
  • Medida Derivada (Derived Measure): Se calcula a partir de dos o más medidas base (ej: densidad de fallos = número de fallos / líneas de código).
  • Indicador (Indicator): Representa el nivel de logro de un requisito o característica de calidad. Combina valores de medidas (ej: "El sistema alcanza el 99.5% de disponibilidad").

Demostración Visual (Texto): Relación Medida - Indicador - Característica

Una medida por sí sola no nos dice si la calidad es buena o mala. Necesitamos un indicador y un valor de referencia para evaluarla.

Medida Base: Número de horas de operación antes de un fallo (Tiempo medio entre fallos - MTBF)
   ↓
Medida Base: Número total de horas de operación
   ↓
Medida Derivada: MTBF (Horas de operación / Número de fallos) -> Si tuviéramos otra medida "Número de fallos"
   ↓
Indicador (Relacionado a Fiabilidad - Disponibilidad): "El sistema tiene un MTBF de 500 horas"
   ↓
Evaluación: ¿Es 500 horas un buen MTBF? Depende del contexto y los requisitos.

Las medidas nos dan datos crudos; los indicadores nos dan contexto para evaluar la calidad.

Paso 5: Aplicando ISO/IEC 25010 y 2502x

En un nivel intermedio, la aplicación implica usar estos modelos para:

  1. Especificar Requisitos de Calidad: No solo funcionales, sino también de calidad. Por ejemplo: "El sistema debe tener una Disponibilidad del 99.9%" (Fiabilidad - Disponibilidad, con un Indicador y valor).
  2. Planificar la Evaluación: Decidir qué características se evaluarán, qué medidas se usarán y qué valores se considerarán aceptables.
  3. Medir y Evaluar: Recopilar datos (medidas) y analizarlos usando indicadores para determinar si se cumplen los requisitos de calidad.

Demostración Visual (Texto): Proceso Simplificado de Evaluación

Cómo se conecta el modelo de calidad y la medición:

Necesidad/Requisito -> Especificar CALIDAD (usando 25010)
   (Ej: El sistema debe ser rápido)
         ↓
Elige CARACTERÍSTICA(S) (Ej: Eficiencia de Desempeño)
         ↓
Elige SUB-CARACTERÍSTICA(S) (Ej: Comportamiento Temporal)
         ↓
Define REQUISITO MEDIBLE (Ej: El tiempo de respuesta para la búsqueda debe ser < 3 segundos)
         ↓
Define MEDIDAS (Ej: Tiempo transcurrido desde la solicitud hasta la respuesta)
         ↓
Realiza MEDICIÓN (Ej: Ejecutar 100 búsquedas, registrar tiempos)
         ↓
Calcula INDICADOR (Ej: Tiempo promedio de respuesta = 2.5 segundos)
         ↓
EVALÚA (Ej: 2.5s < 3s -> Requisito de tiempo de respuesta CUMPLIDO)

Este proceso sistemático nos permite pasar de requisitos cualitativos ("rápido") a evaluaciones cuantitativas ("tiempo promedio de 2.5s, requisito cumplido").
